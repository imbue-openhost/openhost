## Minor Issues

1. **`rename_app` PRAGMA FK toggle relies on SQLAlchemy's lazy-BEGIN, with no test coverage** — `compute_space/web/routes/api/apps.py:450-469`. `await session.execute(text("PRAGMA foreign_keys=OFF"))` is issued *as the first statement after* `await session.commit()` on line 436. Verified empirically that this works: SQLAlchemy only emits `BEGIN` on the next DML, so the PRAGMA lands outside a txn and takes effect. But if anyone adds any statement between the commit and the PRAGMA OFF, the PRAGMA becomes a no-op (confirmed: `PRAGMA foreign_keys=OFF` inside an active txn silently fails), and the subsequent `UPDATE apps SET name=…` will trip the FK `ON DELETE CASCADE` check (child tables `AppDatabase`, `AppPortMapping`, `AppToken`, `ServiceProvider`, `Permission` all FK to `apps.name`, no `ON UPDATE CASCADE`). There is no test for `rename_app`, so this would surface only in production. **Suggested fix**: either (a) declare `ON UPDATE CASCADE` on the five FKs and drop the PRAGMA toggle entirely, or (b) run the rename via `async with engine.connect() as conn: await conn.execution_options(isolation_level="AUTOCOMMIT")…` so the PRAGMA is guaranteed outside a txn. Option (a) is simpler and removes the footgun.

2. **`test_ports.py` was rewritten, not left as-is** — `compute_space/tests/test_ports.py` diff is effectively a full rewrite (sync sqlite3 fixture → `pytest_asyncio` async in-memory-engine fixture; every test became `async def` with new `session` fixture). REQ-TEST-3 explicitly says raw-SQL tests "MAY be left as-is initially; rewriting them to use SQLAlchemy is deferred to a follow-up". This is allowed (it's a *may*, and the behavioural coverage is preserved), but it overlaps scope with the deferred follow-up marker set up in REQ-TEST-4. Worth either keeping the rewrite (and noting that the test-rewrite follow-up now only covers the remaining raw-SQL tests like `test_storage.py::_init_apps_db` and `test_migrations.py`), or reverting `test_ports.py` to use `check_port_available(port, raw_sqlite_conn)` via a tiny sync shim. Preference: keep the rewrite but flag it in the handoff so the follow-up agent doesn't redo this work.

3. **In-memory test fixture in `test_ports.py` does not attach `_enable_sqlite_pragmas`** — `test_ports.py:22` builds a bare `create_async_engine("sqlite+aiosqlite:///:memory:")` without the WAL/FK listener. Irrelevant for `:memory:` (WAL is moot, and the tests don't exercise FK CASCADE), but it establishes a pattern where future tests adding on-disk fixtures might forget the listener and silently run without WAL. Consider factoring a `_test_engine(db_url)` helper in `tests/conftest.py` that attaches the listener for parity.

## Suggestions (non-blocking)

- **N+1 in `permissions.grant_permissions` / `permissions.revoke_permissions`** — `core/permissions.py:43-47, 55-61` loop `session.execute` per key. Both can be collapsed to a single `sqlite_insert(Permission).values([...]).on_conflict_do_nothing(...)` / `delete(Permission).where(Permission.consumer_app == app, Permission.permission_key.in_(keys))`. Typical key lists are small (≤10) so perf is not a concern — the rewrite is purely for clarity.
- **Async engine shared across event loops** — `init_engine` creates a single process-wide `AsyncEngine` with the default pool. `deploy_app_background`, `reload_app_background`, `_check_app_status`, `_restart_apps_sequential`, and `enforce_storage_guard` all call `asyncio.run(...)` inside background threads, each of which opens a session against that shared engine. aiosqlite's per-connection worker thread is event-loop-bound; reusing a pooled connection across loops is a known SQLAlchemy footgun and can cause hangs under load. Tests pass today (each run is short-lived, the pool is small). Safer alternatives: use `poolclass=NullPool` on the async engine (one-shot connections), or create a fresh engine per background thread. Worth tracking as a follow-up, not a blocker.
- **`_check_app_status_async` / `_stop_running_apps` commit cadence** — both loop through rows and call `await session.commit()` either at the very end (`_check_app_status_async`) or once per iteration (`_stop_running_apps:266`). The latter is inside a loop and commits each `UPDATE App…status=error` individually — fine for correctness but the commits happen per-app which fragments the txn. Minor.
- **`expire_on_commit=False`** on `async_sessionmaker` is the right call for request-scoped async sessions, but it means ORM instances can stay stale if another session writes between a read and a later access. Today the code is single-writer-per-resource so this is safe; if that ever changes, a few call sites (`core/apps.py::start_app_process` reading `app_row.docker_container_id` after a `to_thread(build_image)` round-trip) could surprise someone.
- **Orphan `insert_and_deploy` background-thread race** — `core/apps.py:324` launches `threading.Thread(target=deploy_app_background, …)` immediately after `await session.commit()`. The thread's fresh `get_session_maker()()` session will see the committed `App`/`AppPortMapping`/`AppToken` rows. Fine today, but worth a comment at the call site to explain why the commit must precede the thread launch.
- **Handoff SHA** — please tag or record the CI-passing SHA (per REQ-TEST-4) in the PR description or as a lightweight `sculptor/agent_docs/sqlalchemy-alembic/phase2-handoff.md` so the follow-up test-rewrite agent has an unambiguous marker.
