## Pit Traps Checked

- **BEGIN TRANSACTION / COMMIT in dump vs `executescript`**: dumps include both (`_snapshot_harness.py:166, 182`). `materialize_state` (`_snapshot_harness.py:238-251`) creates a fresh sqlite connection (autocommit) and calls `executescript(at_sql)` — `executescript` auto-commits any open transaction before running the script, so embedded BEGIN/COMMIT is fine. Confirmed by passing tests loading `at_0001.sql` then walking to `0002`.
- **`CREATE TABLE` without `IF NOT EXISTS`**: dumps emit DDL straight from `sqlite_master.sql` (no rewriting). `materialize_state` guarantees loading only into empty files by `os.remove(db_path)` first (`_snapshot_harness.py:241-244`). Safe.
- **`_yoyo_migration` tracking rows**: dumps **exclude** them — `_application_objects` / `_application_tables` filter `_yoyo_%` / `yoyo_%` (`_snapshot_harness.py:131-132, 145-146`). Consequence: when the harness loads `at_0001.sql` and then applies 0002, yoyo's `to_apply` sees no tracking rows and returns all migrations; `.filter(lambda m: m.id == migration_id)` (`_snapshot_harness.py:64`) narrows to just 0002, and 0001's DDL is never re-executed. **Verified empirically**: `0001->0002` case passed with the synthetic 0002 migration. Caveat: `_yoyo_migration` ends up containing only the newly-applied id, not 0001. This is harmless for pure-DDL migrations but would bite a hypothetical migration that branches on "has 0001 run?" — worth a code comment, not a blocker.
- **Diff reporting**: mismatch path uses `difflib.unified_diff` with labelled fromfile/tofile, wrapped in `pytest.fail(...)` (`test_yoyo_migrations.py:241-253`). Readable.

## Minor

- `_snapshot_harness.py:133-134` — redundant WHERE clauses: `name NOT LIKE '_yoyo_%'` is already checked earlier in the same query, then repeated with `tbl_name NOT LIKE '_yoyo_%'`. The `tbl_name` checks are the load-bearing ones for indexes whose parent table is `_yoyo_migration`; the `name` repeat is redundant. Trivial.
- Test id format in `_case_id` uses `->` (`scenario_port_mappings-empty->0001`) rather than the task spec's `-` separator (`scenario_port_mappings-empty-0001`). Arguably more readable; not worth changing.
- `_snapshot_harness.py:64` — `.filter(lambda m: m.id == migration_id)` after `to_apply` means only the exact migration being stepped gets its tracking row. Add a one-line comment explaining that any later migration whose conditional logic depends on earlier yoyo state will not see a correct `_yoyo_migration` when the harness walks partial ranges.

## Suggestions

- Consider adding a single sanity test that loads `at_NNNN.sql` and verifies `_yoyo_migration` contains the applied id(s) after the walk — documents the tracking-row behavior in executable form.
